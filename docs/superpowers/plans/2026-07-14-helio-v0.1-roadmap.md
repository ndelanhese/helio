# Helio v0.1 Roadmap

Spec source: `docs/superpowers/specs/2026-07-14-helio-local-solar-monitor-design.md`

Implementation is split because protocol, durable core, product UI, analysis, and release packaging have distinct review and verification gates. Execute in this order:

1. `2026-07-14-helio-foundation-protocol.md` — executable Go/React shell plus safe read-only Solarman/SOFAR transport.
2. `2026-07-14-helio-core-data-auth-api.md` — SQLite, collector, authenticated bootstrap, REST, SSE, and onboarding.
3. `2026-07-14-helio-responsive-ui.md` — approved Solar editorial interface for Now, History, Insights, and Settings.
4. `2026-07-14-helio-analysis-alerts.md` — weather cache, expectation model, confidence, and internal alerts.
5. `2026-07-14-helio-container-release.md` — production image, PWA, CI, backup, operations docs, and v0.1 release gate.

Each plan inherits these invariants:

- No Modbus write function, endpoint, command, or register allowlist exists in v0.1.
- Deployment IP, logger serial, location, tariff, and credentials never enter source control or test fixtures.
- PV2 may be configured inactive and must never create a fault or underproduction penalty.
- Missing telemetry remains a gap; no interpolation is persisted as measured production.
- `/health/live` checks process only; `/health/ready` checks HTTP/database; `/health/components` reports logger/weather degradation.
- Port `8080`, database `/data/helio.db`, one runtime process, one runtime container, one persistent volume.
